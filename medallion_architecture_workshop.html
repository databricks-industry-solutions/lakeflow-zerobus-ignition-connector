2. Open Table Formats

15 minutes

OPEN FORMATS

Starting Point: Parquet Files

What is Parquet?

  • Open-source columnar file format
  • Stores data by column, not by row
  • Built-in compression (Snappy, ZSTD)
  • Self-describing — schema is in the file
  • Language-agnostic (Java, Python, Rust)

Why columnar?

  • Analytics reads few columns, many rows
  • SELECT AVG(temperature) reads one column
  • Skips everything else — massive I/O savings
  • Compression is better within one data type
Parquet is the foundation. But a pile of Parquet files isn't a table — it's just files.

DELTA LAKE

Parquet + Transaction Log

raw_tags/ _delta_log/ ← the brain 00000000000000000000.json ← version 0 00000000000000000001.json ← version 1 00000000000000000002.json ← version 2 00000000000000000010.checkpoint.parquet part-00000-abc123.parquet ← data part-00001-def456.parquet ← data part-00002-ghi789.parquet ← data

Two parts:

  • _delta_log/ — JSON files recording every transaction
  • *.parquet — The actual data files

Every write creates:

  • New Parquet file(s) with the data
  • New JSON entry in _delta_log declaring file membership

FILE STRUCTURE

Inside a Transaction Log Entry

{
  "commitInfo": {
    "timestamp": 1712700000000,
    "operation": "WRITE",
    "operationParameters": { "mode": "Append" }
  }
}
{
  "add": {
    "path": "part-00000-abc123.snappy.parquet",
    "size": 48291,
    "modificationTime": 1712700000000,
    "dataChange": true,
    "stats": "{\"numRecords\":1024,
              \"minValues\":{\"event_time\":1712699000},
              \"maxValues\":{\"event_time\":1712700000}}"
  }
}
  • commitInfo — who did what, when
  • add / remove — which Parquet files changed
  • stats — min/max per column (enables data skipping)
11
DELTA LAKE

Time Travel

How it works

  • Version 0: Created table (empty)
  • Version 1: Added file A
  • Version 2: Added files B, C
  • Version 3: Removed A, added D

Current state (replay 0→3): files B, C, D

Version 2 (replay 0→2): files A, B, C

In SQL

-- Current data
SELECT * FROM raw_tags;

-- Data as of yesterday
SELECT * FROM raw_tags
  TIMESTAMP AS OF '2026-04-09';

-- A specific version
SELECT * FROM raw_tags
  VERSION AS OF 42;

-- Undo a bad write
RESTORE TABLE raw_tags
  TO VERSION AS OF 41;

4. The Medallion Architecture

10 minutes

MEDALLION PATTERN

Three Layers, One Purpose

Bronze

"Land everything, trust nothing"

Raw data as-is. Append-only. No transforms. The system of record.

↓ clean, validate, parse ↓

Silver

"Cleaned, validated, enriched"

Deduped. Schema enforced. Business keys extracted. Quality rules applied.

↓ aggregate, model, serve ↓

Gold

"Ready for the business question"

Pre-aggregated for dashboards. One row per entity. CEO-readable.

Section 5

Bronze

15 minutes

BRONZE LAYER

Bronze Principles

  • 1 Append-only — never update or delete raw data
  • 2 Source schema — keep original column names and types
  • 3 Add metadata — ingestion timestamp, source system, file name
  • 4 Multiple landing patterns — streaming via Zerobus, batch via Auto Loader (CSV/JSON/Parquet)
  • 5 Enable Change Data Feed — downstream reads changes incrementally
  • 6 No business logic — that's silver's job
Bronze is your insurance policy. When silver logic has a bug, you replay from bronze.
22
BRONZE LAYER

Our Bronze Table: raw_tags

CREATE TABLE medallion.bronze.raw_tags (
  event_id          STRING      NOT NULL,   -- UUID per event
  event_time        BIGINT      NOT NULL,   -- microseconds since epoch
  tag_path          STRING      NOT NULL,   -- packed path from source
  tag_provider      STRING      NOT NULL,   -- e.g. "agl_bess"
  numeric_value     DOUBLE,                 -- sensor reading
  string_value      STRING,                 -- text values
  quality           STRING,                 -- "GOOD", "BAD", "UNCERTAIN"
  quality_code      INT,                    -- OPC quality code
  source_system     STRING      NOT NULL,   -- which gateway sent this
  ingestion_timestamp BIGINT               -- when we received it
)
LOCATION 'abfss://medallion@storage.dfs.core.windows.net/bronze/raw_tags'
TBLPROPERTIES (
  'delta.enableChangeDataFeed' = 'true'
);
Schema-on-read: event_time stored as BIGINT, tag_path as an opaque string. We don't interpret them yet — that's silver's job.

Silver

20 minutes

SILVER LAYER

Silver = Schema-on-Write

Bronze: Schema-on-Read

  • Store data in source format
  • Interpret when you read it
  • BIGINT timestamps, packed strings
  • No validation at write time

Silver: Schema-on-Write

  • Enforce types before storing
  • Validate every row on write
  • Proper TIMESTAMP, extracted keys
  • Bad data caught, not hoped away
Bronze preserves fidelity. Silver enforces trust. The boundary between them is where schema-on-read becomes schema-on-write.

  lower(concat(location, '_', asset))       AS asset_id,

  COALESCE(numeric_value, string_value)     AS tag_value
FROM medallion.bronze.raw_tags

SILVER LAYER

Silver Step 2: Aggregate

-- 1-minute tumbling window aggregation
CREATE OR REFRESH STREAMING TABLE
  medallion.silver.aggregated_tags
AS SELECT
  window(event_timestamp, '1 minute').start
    AS window_start,
  window(event_timestamp, '1 minute').end
    AS window_end,
  tag_name, asset_id,
  AVG(tag_value)     AS avg_value,
  MIN(tag_value)     AS min_value,
  MAX(tag_value)     AS max_value,
  STDDEV(tag_value)  AS stddev_value,
  COUNT(*)           AS sample_count
FROM STREAM(medallion.silver.parsed_tags)
GROUP BY
  window(event_timestamp, '1 minute'),
  tag_name, asset_id;
99%
volume reduction
Bronze: 162,000 rows/min
Silver: ~1,350 rows/min

DATA QUALITY

Silver Step 3: Data Quality

Lakeflow Pipeline Expectations

valid_event_timestamp — event_timestamp IS NOT NULL EXPECT (warn)
has_asset_id — asset_id IS NOT NULL AND asset_id != '' EXPECT (warn)
valid_quality_code — quality_code BETWEEN 0 AND 255 EXPECT (warn)
not_future_event — event_timestamp <= now() + INTERVAL 5 MINUTES EXPECT OR DROP
Three modes: EXPECT (warn but keep) • EXPECT OR DROP (silently remove) • EXPECT OR FAIL (stop the pipeline)
Production pattern — quarantine tables: Route bad rows to a sibling _quarantine table for investigation instead of dropping. Data isn't lost — it's isolated. We use DROP today for simplicity; in production you'd quarantine.

SILVER LAYER

Silver Step 4: Enrich

-- Stream-static join: streaming data + static reference table
CREATE OR REFRESH STREAMING TABLE medallion.silver.enriched_tags
AS SELECT
  a.window_start, a.window_end,
  a.tag_name, a.asset_id,
  a.avg_value, a.min_value, a.max_value, a.sample_count,
  -- From reference table:
  m.signal_name,       -- e.g. "Rack Temperature"
  m.unit,              -- e.g. "°C"
  m.source_domain      -- e.g. "battery_thermal"
FROM STREAM(medallion.silver.aggregated_tags) a
LEFT JOIN medallion.silver.signal_mapping m
  ON a.tag_name = m.tag_path;
Before: tag_name = "thermal/rack_temp_c"
After: signal_name = "Rack Temperature", unit = "°C", domain = "battery_thermal"

Gold

GOLD LAYER

Gold Principles

  • 1 Business grain — one row per entity per time period
  • 2 Sources merge here — silver stays source-aligned; gold joins across
  • 3 Named for the question — health_scores, not agg_enriched_v3_final
  • 4 Pre-computed — no complex joins at query time
  • 5 Consumer-ready — a dashboard can SELECT * and go
Gold's contract: "Ask your question, get your answer. No SQL PhD required."
36
GOLD PATTERNS

Two Common Patterns

Star Schema Traditional BI

  • Fact + dimension tables
  • Optimized for slice-and-dice queries
  • Historical reporting, dimensional analysis

Materialized Views Real-time Ops

  • Pre-aggregated, auto-refreshed
  • Optimized for current state dashboards
  • Operational monitoring, alerting, ML features
Today we use materialized views — operations teams need current state, not historical drill-downs. Choose the pattern that matches your consumer.

HEALTH SCORING

Gold Table: health_scores

CREATE OR REFRESH MATERIALIZED VIEW medallion.gold.health_scores
AS SELECT
  now()                                    AS scored_at,
  asset_id,
  1.0 - (anomalous_tag_count / total_key_tags)  AS health_score,
  first_anomalous_tag                      AS primary_risk_tag,
  risk_description,
  estimated_hours_to_failure
FROM silver_health_features;

REVENUE RISK

Gold Table: revenue_risk

CREATE OR REFRESH MATERIALIZED VIEW
  medallion.gold.revenue_risk
AS SELECT
  h.asset_id,
  p.risk_window_start,
  p.forecast_price_aud_mwh,
  a.capacity_mw,
  h.health_score,
  (1 - h.health_score) AS trip_probability,
  a.capacity_mw * hours * price
    * (1 - h.health_score)
    AS revenue_at_risk_aud
FROM medallion.gold.health_scores h
JOIN medallion.gold.price_forecast p
  ON p.price > 300
JOIN medallion.silver.asset_registry a
  ON a.asset_id = h.asset_id;

The business question:

"During the next high-price window, how much revenue are we risking if an unhealthy asset trips?"

health_score × price forecast × capacity = dollars at risk

DATA PRODUCTS

Gold Tables Are Data Products

A data product has an owner, a contract, an SLA, and is discoverable. Your gold tables already have most of this.

CONTRACT

Schema enforced by pipeline. Quality rules via expectations. Column types are the API.

DISCOVERABLE

COMMENT ON TABLE + column descriptions in Unity Catalog. Searchable by Genie and humans.

GOVERNED

UC access controls, lineage from source to dashboard, audit trail via Delta time travel.

AGL enterprise pattern: Source-aligned products (bronze/silver per domain) → Consumer-aligned products (gold for specific audiences). health_scores and revenue_risk are consumer-aligned data products.
